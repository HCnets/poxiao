## 总体架构

### 6.1 架构概述

客户端采用“单模块应用 + 多业务页面 + 本地存储 + 外部网关接入”的结构形态。`MainActivity` 仅负责承载 Compose 应用入口，页面导航、业务编排和本地状态管理主要由 `PoxiaoApp()` 统一协调。

从分层角度，系统可划分为四层：

1. 表现层：Compose 页面、导航、主题、组件、交互状态
2. 业务层：待办、复习计划、课程笔记、导出、本地提醒等逻辑
3. 接入层：HITA 教务网关、Spark 智能体网关、Provider 工厂
4. 存储层：SharedPreferences、文件附件目录、AndroidKeyStore 加密存储

## 接口定义

### 8.1 接口分类

本项目接口分为三类：

1. 外部网络接口：HITA、Spark
2. 应用内业务接口：Repository、Gateway、Store
3. 本地导入导出接口：本地备份、附件导入、通知调度

表 8-1 接口总览表。表头说明：协议用于识别链路形态；实现状态用于区分已上线、预留、仅预览三类状态。

| 接口标识 | 主要方法/能力 | 协议/载体 | 主要实现 | 实现状态 |
| --- | --- | --- | --- | --- |
| `HiagentGateway` | `sendText()`、`sendVoice()`、`sendImage()` | 应用内契约 | `PreviewHiagentGateway`、`SparkAssistantGateway`、`DeepSeekAssistantGateway` | 文本已实现，语音/图片预留 |
| `HitaScheduleGateway` | `login()`、`fetchTerms()`、`fetchWeeks()`、`fetchWeekSchedule()`、`fetchDaySchedule()`、`fetchGrades()`、`fetchTeachingBuildings()`、`fetchEmptyClassrooms()` | HTTPS/JSON/表单 | `HitaScheduleGatewayImpl` | 已实现 |
| 本地备份接口 | `buildLocalBackupPayload()`、`restoreLocalBackupPayload()` | JSON 文本 | `PoxiaoApp.kt` 内部函数 | 已实现 |
| `AcademicGateway` | `importFromHitas()`、`getEmptyClassrooms()`、`getGrades()` | 应用内契约 | `PreviewAcademicGateway`、`HitaAcademicGateway` | 已实现 |

统一约束如下：

- 接口命名以“能力 + 动作”表达，不使用含糊简称
- 外部接口优先定义 Gateway 契约，再由 Repository 或 UI 调用
- 对用户可见的失败结果必须返回明确文案，不得直接透出原始堆栈
- 所有示例数据均为脱敏或演示值，不得复用真实账号、令牌、密钥

### 8.2 智能体接口定义

#### 8.2.1 Provider 抽象接口

接口名称：`HiagentGateway.sendText()`  
接口类型：应用内统一网关接口  
调用方：首页智能体模块  
实现方：`PreviewHiagentGateway`、`SparkAssistantGateway`、`DeepSeekAssistantGateway`

表 8-2 智能体文本接口字段说明。表头说明：字段类型以 Kotlin 类型为准，请求与响应为应用内对象契约。

| 字段 | 方向 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| `message` | 请求 | `String` | 是 | 用户输入文本 |
| `history` | 请求 | `List<ChatMessage>` | 否 | 最近历史消息，默认空列表 |
| `id` | 响应 | `String` | 是 | 消息唯一标识 |
| `role` | 响应 | `String` | 是 | 角色，固定为 `assistant` |
| `content` | 响应 | `String` | 是 | 返回文本 |
| `timestamp` | 响应 | `Long` | 是 | 毫秒时间戳 |

请求示例：

```json
{
  "message": "请根据本周课表帮我安排复习计划",
  "history": [
    {
      "id": "msg-user-001",
      "role": "user",
      "content": "我下周有考试",
      "timestamp": 1766622000000
    }
  ]
}
```

响应示例：

```json
{
  "id": "assistant-7bb6f7e5",
  "role": "assistant",
  "content": "已根据你的课表与待办情况生成复习建议。",
  "timestamp": 1766622003320
}
```

状态码约定：

- 应用内接口不直接返回 HTTP 状态码，统一通过业务响应文本和异常兜底表达
- 若底层网络失败，统一归一为“服务暂时不可用”文案

错误码建议：

表 8-3 智能体错误码建议表。表头说明：当前工程未形成统一错误码枚举，以下为维护阶段建议标准。

| 错误码 | 说明 | 当前触发条件 | 重试策略 |
| --- | --- | --- | --- |
| `AST-001` | 请求为空 | 输入文本为空白 | 不重试，前端提示用户重新输入 |
| `AST-002` | Provider 未配置 | `SPARK` 配置字段缺失 | 不自动重试，提示补充配置 |

#### 8.2.2 Spark WebSocket 接口

接口名称：Spark Assistant WebSocket  
协议：`WSS`  
认证方式：URL 签名参数 + Header 参与签名  
调用路径：`wss://spark-openapi.cn-huabei-1.xf-yun.com/v1/assistants/{assistantId}`

签名参数说明：

- `authorization`
- `date`
- `host`

请求载荷示例：

```json
{
  "header": {
    "app_id": "demo_app_id",
    "uid": "android_a1b2c3d4e5f6"
  },
  "parameter": {
    "chat": {
      "domain": "generalv3"
    }
  },
  "payload": {
    "message": {
      "text": [
        {
          "role": "user",
          "content": "请总结今天课表重点"
        }
      ]
    }
  }
}
```

响应载荷示例：

```json
{
  "header": {
    "code": 0,
    "message": "success"
  },
  "payload": {
    "choices": {
      "status": 2,
      "text": [
        {
          "role": "assistant",
          "content": "今天下午有机器学习课程，建议提前复习上一讲内容。"
        }
      ]
    }
  }
}
```

状态码：

- `0`：成功
- 非 `0`：失败，读取 `header.message`

重试策略：

- 首次使用 `generalv3`
- 若失败，回退一次到 `general`
- 若仍失败，返回统一失败文本

### 8.3 HITA 教务接口定义

#### 8.3.1 认证与会话初始化

接口集合：

- `initializeRoute()`
- `fetchRsaKey()`
- `login(studentId, password)`

认证说明：

- 首先调用 `InitRoute` 获取路由上下文
- 再调用 `FetchRsa` 拉取 RSA 密钥信息
- 最后调用 `LdapLogin` 发起登录
- 登录成功后获取 `access_token`、`refresh_token`、路由 Cookie 与会话信息

表 8-5 HITA 登录请求字段说明。表头说明：字段名称以服务端表单参数和返回 JSON 为准。

| 字段 | 方向 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| `username` | 请求 | `String` | 是 | 学号 |
| `password` | 请求 | `String` | 是 | 密码，明文或 RSA 加密结果 |
| `access_token` | 响应 | `String` | 是 | 教务访问令牌 |
| `refresh_token` | 响应 | `String` | 否 | 刷新令牌 |
| `yhdm` | 响应 | `String` | 否 | 用户代码，通常为学号 |
| `yhxm` | 响应 | `String` | 否 | 学生姓名 |
| `pylx` | 响应 | `String` | 否 | 学生类别 |

请求示例：

```http
POST /authentication/ldap HTTP/1.1
Content-Type: application/x-www-form-urlencoded

username=2026123456&password=****** 
```

响应示例：

```json
{
  "access_token": "eyJhbGciOi...",
  "refresh_token": "refresh_demo_token",
  "info": {
    "yhdm": "2026123456"
  },
  "data": {
    "yhxm": "张同学",
    "pylx": "1"
  }
}
```

状态码：

- `200`：请求成功
- `401/403`：认证失败或会话失效
- `5xx`：教务侧服务错误

错误码建议：

| 错误码 | 说明 | 触发条件 | 重试策略 |
| --- | --- | --- | --- |
| `HITA-001` | 登录失败 | 学号或密码错误 | 不自动重试 |
| `HITA-002` | 会话失效 | Token 过期、401、403 | 引导重新登录 |
| `HITA-003` | RSA 获取失败 | RSA 接口异常 | 最多重试 1 次 |
| `HITA-004` | 课表查询失败 | 周课表或日课表接口失败 | 间隔 2 秒重试 1 次 |
| `HITA-005` | 成绩查询失败 | 成绩接口失败 | 不自动重试，允许用户手动刷新 |

表 8-6 HITA 登录状态码表。表头说明：HITA 侧未提供统一业务码枚举时，以 HTTP 状态和响应字段共同判定。

## 安全策略

### 10.1 认证方式

客户端当前涉及两类认证：

1. HITA 教务认证：用户名密码登录，换取访问令牌
2. Spark 服务认证：签名 URL 参数 + `appId/apiKey/apiSecret`

认证要求：

- 教务密码不得以明文长期持久化
- 智能体密钥不得以默认真实值随安装包发出
- 发布版必须通过环境变量或私有配置注入敏感字段

### 10.2 权限粒度

当前 Android 权限粒度如下：

表 10-1 Android 权限表。表头说明：申请时机必须最小化，禁止一次性无差别申请。

| 权限 | 用途 | 授权时机 | 失败降级策略 |
| --- | --- | --- | --- |
| `INTERNET` | 网络请求 | 安装即声明 | 无网络能力时展示离线提示 |
| `POST_NOTIFICATIONS` | 本地提醒通知 | 用户进入通知相关功能时 | 保持功能可用，仅不发送通知 |
| `SCHEDULE_EXACT_ALARM` | 精准提醒 | 需要精准提醒时 | 回退为普通提醒策略 |

### 10.3 密钥管理

密钥管理规范：

1. `SPARK_ASSISTANT_APP_ID`、`SPARK_ASSISTANT_API_KEY`、`SPARK_ASSISTANT_API_SECRET` 默认值必须为空
2. 仅允许通过环境变量、私有 `gradle.properties` 或 CI Secret 注入
3. 禁止将真实密钥提交至 Git 仓库
4. 若需多人协作，统一通过密钥分发平台或受控文档传递

密钥生命周期要求：

- 生成：由项目负责人或安全管理员生成并登记用途
- 存放：仅存于本机私有配置、CI Secret 或受控密码库
- 使用：开发、测试、生产环境分离，禁止跨环境复用
- 轮换：发生泄露、离职交接、供应商切换时立即轮换
- 注销：废弃密钥须同步清理本地、CI 与归档文档中的历史引用

### 10.4 本地敏感信息保护

当前项目使用 `SecurePrefs` 对敏感信息进行加密存储：

- 算法：`AES/GCM/NoPadding`
- 密钥来源：`AndroidKeyStore`
- 密钥长度：`256`
- IV：每次加密随机生成
- 存储前缀：`enc_v1:`

高敏字段：

- 学号
- 教务密码
- 后续若引入 Token 持久化，应同样纳入加密

### 10.5 审计日志规范

当前工程尚未形成独立审计日志子系统。维护阶段要求新增或补齐以下日志格式规范：

表 10-2 审计日志格式表。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `event_time` | `String` | 是 | ISO 8601 时间 |
| `event_type` | `String` | 是 | 事件类型，如 `LOGIN_SUCCESS` |
| `module` | `String` | 是 | 模块名，如 `HITA`、`ASSISTANT` |
| `operator_id` | `String` | 否 | 脱敏后的学号或设备标识 |
| `result` | `String` | 是 | `SUCCESS`、`FAIL` |
| `error_code` | `String` | 否 | 错误码 |
| `trace_id` | `String` | 否 | 跟踪 ID |
| `detail` | `String` | 否 | 脱敏摘要 |

审计日志补充规则：

- 日志级别至少区分 `INFO`、`WARN`、`ERROR`
- 不得记录明文学号全量、密码、完整令牌、完整签名串
- `operator_id` 必须脱敏，如 `2026****56`
- 审计日志保留期建议不少于 180 天
- 若接入第三方埋点平台，需沿用同一事件命名和脱敏规则

日志示例：

```json
{
  "event_time": "2026-04-28T21:30:11+08:00",
  "event_type": "HITA_LOGIN_FAIL",
  "module": "HITA",
  "operator_id": "2026****56",
  "result": "FAIL",
  "error_code": "HITA-001",
  "trace_id": "trace-2f5b7f86",
  "detail": "账号或密码不正确"
}
```

### 10.6 安全评审要求

安全评审至少覆盖以下内容：

- 密钥未硬编码
- 敏感数据已加密或仅驻留内存
- 备份文本脱敏边界明确
- 附件导入不越权访问外部目录
- FileProvider 暴露路径最小化

| 指标 | 目标值 | 采集方式 | 告警阈值 |

- `Poxiao/Reminder`

## 17. 版本迭代与多校适配规范 (v1.1.0 增补)

### 17.1 语义化版本管理 (SemVer)

项目统一采用 `X.Y.Z` 格式进行版本管理：

1.  **主版本号 (X)**：
    - 触发条件：重大架构变更（如引入多智能体协同、支持多校平台化）、品牌重构、底层协议完全不兼容。
    - 示例：`1.0.0` -> `2.0.0`。
2.  **次版本号 (Y)**：
    - 触发条件：新增独立功能模块、显著增强现有功能（如本轮全域联动）、接口向后兼容的扩展。
    - 示例：`1.0.0` -> `1.1.0`。
3.  **修订号 (Z)**：
    - 触发条件：紧急 Bug 修复、UI/UX 细节优化、文案调整。
    - 示例：`1.1.0` -> `1.1.1`。

### 17.2 Git 标签与分支规范

-   **标签 (Tag)**：每次 `Y` 位或 `X` 位版本发布必须打 Tag，格式为 `vX.Y.Z-release_name`。
-   **分支 (Branch)**：
    - `main`：生产稳定分支，仅接受已验证的 Merge Request。
    - `feature/*`：功能开发分支。
    - `archive/*`：重大重构前的基线归档分支。

### 17.3 多校适配演进方案 (Multi-School)

针对未来可能出现的“非哈深教务”版本，采用以下技术路线：

#### 1. Gradle Product Flavors (多渠道构建)
利用 Gradle 的变体管理机制，在不增加代码冗余的前提下区分学校：
```kotlin
android {
    productFlavors {
        create("hitsz") {
            dimension = "school"
            applicationIdSuffix = ".hitsz"
            versionNameSuffix = "-hitsz"
        }
        create("generic") {
            dimension = "school"
            applicationIdSuffix = ".generic"
            versionNameSuffix = "-generic"
        }
    }
}
```

#### 2. 接口抽象化 (Interface Abstraction)
将教务服务（HITA）抽象为通用接口 `AcademicService`，不同学校实现各自的逻辑：
- `hitsz` 渠道：实现 `HitaAcademicServiceImpl`。
- `generic` 渠道：实现基于通用教务系统的 `GenericAcademicServiceImpl`。

#### 3. 资源目录隔离 (SourceSet Isolation)
利用 `src/hitsz` 和 `src/generic` 目录隔离特定学校的 UI 资源（如校徽、主题色、特定教务入口），而 `src/main` 保持核心逻辑共享。

#### 4. 智能体自适应
智能体根据当前 Flavor 自动切换 Prompt 模板和上下文注入逻辑，确保在不同学校环境下提供准确的日程建议。
