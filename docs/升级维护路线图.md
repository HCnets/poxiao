# 升级维护路线图

## 1. 目标

本路线图服务于 `C:\Users\HCnets\Desktop\AI` 中的 `Poxiao / 南工破晓` 主工程，核心目标不是继续做资料整理，而是让项目进入可持续升级、可持续维护、可持续交付的状态。

当前建议的维护原则：

- 以本地 `AI` 目录为唯一主工程真源
- 优先保证构建稳定、核心功能稳定、配置安全
- 再逐步处理结构拆分、测试补强和能力扩展

## 2. 当前状态判断

### 已具备的基础

- 根工程仍是单模块结构，维护入口集中，适合阶段性升级
- 主工程可以构建运行，关键模块已经落地
- 当前 IDE 诊断没有明显 Kotlin 编译错误

### 当前主要风险

1. 构建配置中仍带有默认的星火密钥和配置值，不适合长期维护或对外分发
2. `DeepSeek` 仍是占位接入，属于功能预留而非可交付能力
3. `PoxiaoApp.kt` 仍承担过多页面和编排职责，维护成本过高
4. 目前没有自动化测试文件，后续升级回归成本偏高
5. 工程仍以本地存储和页面内状态管理为主，模块边界不够清晰

## 3. 已确认的维护事实

### 构建层

- 根工程只有 `:app` 一个主模块
- 当前构建插件版本：
  - AGP：`9.1.0`
  - Kotlin：`2.3.10`
  - Compose Plugin：`2.3.10`
- 当前 Android 基线：
  - `compileSdk = 36`
  - `targetSdk = 36`
  - `minSdk = 31`
  - `JVM target = 17`

### 安全与配置

- `app/build.gradle.kts` 中仍为 `SPARK_ASSISTANT_APP_ID`、`SPARK_ASSISTANT_API_KEY`、`SPARK_ASSISTANT_API_SECRET` 提供了默认值
- 这意味着即使本地未单独配置环境变量，构建产物里仍可能出现真实或半真实凭据

### AI 接入层

- 当前默认链路仍以 `SPARK` 为主
- `DeepSeek` 只完成了 provider 接线和兜底文案，尚未完成真实网络实现
- `DeepSeek` 的语音与图像入口也仍为占位实现

### 结构层

- `MainActivity` 本身很薄，只负责挂载 `PoxiaoApp()`
- 页面与业务编排仍高度集中在 `PoxiaoApp.kt`
- `PoxiaoApp.kt` 已经至少延伸到 10000 行以上，说明首页、导航、桥接逻辑和局部页面仍然耦合较深

### 质量层

- 当前 `app` 目录下未发现 `test` 或 `androidTest` 自动化测试文件
- 这意味着版本升级和重构将主要依赖人工回归

## 4. 维护优先级

### P0：立即处理

1. 配置安全化
2. 构建基线固化
3. 核心模块回归面收拢

### P1：近期处理

1. 大文件拆分
2. 模块边界梳理
3. 核心回归测试补点

### P2：中期处理

1. AI provider 扩展
2. 数据层统一化
3. 视觉层和导航层的长期演进

## 5. 第一阶段：先稳住工程

建议先用 1 到 2 个迭代，把项目从“能跑”升级到“稳定可维护”。

### 5.1 配置安全化

目标：

- 彻底移除构建脚本里的默认敏感值
- 所有密钥只从 `gradle.properties`、环境变量或本地私有配置读取
- 发布版默认不带可用密钥

建议动作：

1. 把 `SPARK_ASSISTANT_*` 默认值改为空值或演示值
2. 在启动时增加配置检查和用户可读错误提示
3. 区分 `debug` 与 `release` 配置行为

完成标准：

- 本地没有私密配置时也能正常构建
- 但联网智能体功能会明确提示配置缺失，而不是隐式带入默认凭据

### 5.2 构建基线固化

目标：

- 确认当前 AGP、Kotlin、Compose、JDK 组合是稳定的
- 避免一边开发一边被工具链随机影响

建议动作：

1. 固定 Android Studio、Gradle Wrapper、JDK 使用要求
2. 补一份最小构建说明
3. 记录当前已验证可用的依赖版本组合

完成标准：

- 新机器拉起项目时，不需要反复猜测工具链版本

### 5.3 核心回归面收拢

目标：

- 为升级和重构先定义最小可接受回归范围

建议首批固定回归模块：

1. 智能体文本链路
2. HITA 登录与课表获取
3. 待办新增、完成、顺延
4. 复习计划同步与转待办
5. 课程笔记附件打开与删除
6. 通知提醒重排与测试通知
7. 环境声播放与切页恢复

## 6. 第二阶段：降低维护成本

### 6.1 拆分 `PoxiaoApp.kt`

这是当前最值得做的结构优化项。

目标：

- 把超大编排文件拆成可独立维护的页面、组件和状态模块

建议拆分顺序：

1. 先拆首页与更多页入口卡片
2. 再拆智能体桥接和复习计划桥接逻辑
3. 最后拆通用组件、导航状态和局部弹层

建议结果形态：

- `ui/home/`
- `ui/more/`
- `ui/navigation/`
- `ui/components/`
- `assistant/bridge/`

完成标准：

- `PoxiaoApp.kt` 只保留应用级导航、主题注入和少量顶层编排

### 6.2 统一状态与存储入口

当前项目已经有不少页面直接使用本地偏好存储和页面内状态。

目标：

- 减少“页面自己读写存储、页面自己做业务决策”的耦合

建议动作：

1. 为待办、复习计划、课程笔记、通知偏好分别建立清晰的 store/repository 边界
2. 把分散的本地存储 key 逐步集中
3. 避免新增页面继续直接拼接存储逻辑

完成标准：

- 页面主要负责展示与交互
- 业务状态和持久化逻辑尽量从 UI 中抽离

### 6.3 抽离共享组件

目标：

- 降低视觉改动时的波及范围

建议动作：

1. 把玻璃卡片、操作胶囊、筛选条、模块标题等共性组件集中
2. 把重复文案样式、色板使用习惯统一
3. 把高频 formatter 和状态提示收口

## 7. 第三阶段：补质量保证

### 7.1 先补小而准的测试

当前项目没有自动化测试，不适合一口气补很多。

建议优先补：

1. `AssistantGatewayFactory` provider 选择逻辑
2. `DeepSeekAssistantConfig` 配置判定逻辑
3. 复习计划日期顺延和转待办的纯逻辑部分
4. 导出文本拼装逻辑

### 7.2 再补高价值集成回归

建议后补：

1. 通知偏好重排的核心调度逻辑
2. 课程笔记附件元数据处理
3. HITA 返回解析和异常兜底

原则：

- 只补高价值测试，不补纯堆数量测试

## 8. 第四阶段：能力扩展

当工程已经稳定后，再考虑继续做功能升级。

### AI 侧

1. 真正完成 `DeepSeek` 接入
2. 统一多 provider 的错误处理、超时处理和能力声明
3. 评估语音、图像入口是否走同一套抽象

### 校园服务侧

1. 强化 HITA 异常处理与缓存策略
2. 评估空教室、成绩、课表数据的统一数据入口

### 视觉与交互侧

1. 继续沿用当前 `backdrop` 能力做二次封装
2. `_miuix_main` 只作为导航和组件参考，不直接并入当前主工程

## 9. 推荐执行顺序

建议按下面顺序推进：

1. 配置安全化
2. 构建基线确认
3. 核心回归清单固定
4. 拆分 `PoxiaoApp.kt`
5. 收口本地存储与业务状态
6. 补 3 到 5 个高价值测试
7. 再考虑 DeepSeek 和其他增量能力

## 10. 当前最值得先做的三件事

如果现在马上开始维护，我建议第一批就做这三件：

1. 去掉 `SPARK` 默认敏感配置，改成安全读取模式
2. 拆 `PoxiaoApp.kt` 的首页/更多页入口层
3. 为 `AssistantGatewayFactory` 和复习计划顺延逻辑补最小测试
