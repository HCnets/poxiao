# 版本说明

## 1. 文档目的

这份文档用于说明当前仓库中“未拆分基线版”和“拆分完成版”的保留方式、适用场景与建议工作流，方便后续开发、回退、对比和交接。

## 2. 当前版本布局

- 主维护线：`main`
- 归档基线分支：`archive-monolithic`
- 未拆分基线标签：`monolithic-baseline`

## 3. 两套版本定义

### 3.1 拆分完成版

- 分支：`main`
- 当前提交：`8f4555f`
- 提交说明：`refactor: stabilize split poxiao app structure`
- 角色：
  - 作为后续开发和维护的主线
  - 保留当前已验证通过的拆分结构
  - 后续功能迭代默认基于这一版继续

当前特点：

- `PoxiaoApp.kt` 已收缩为入口壳层
- `PoxiaoAppScaffold.kt` 已收缩为宿主壳层
- `HomeScreen.kt` 已收缩为状态装配 + 页面壳层
- 共享模型、动作包、builder、component、helper 的边界已显著清晰化
- 用户已确认 Android Studio 中“编译成功，功能正常”

### 3.2 未拆分基线版

- 分支：`archive-monolithic`
- 标签：`monolithic-baseline`
- 基线提交：`7a77a44`
- 提交说明：`revert: restore monolithic poxiao app baseline`
- 角色：
  - 作为拆分前的归档对照版本
  - 用于结构回看、差异比对和应急回退
  - 不建议作为后续日常开发主线

当前特点：

- 结构更接近拆分前的单文件 / 大文件组织方式
- 适合做“前后结构差异”对照
- 适合排查“某逻辑原来在哪个大文件里”

## 4. 建议工作流

### 4.1 日常开发

默认使用：

- `main`

原因：

- 当前这条线已经经过本轮真实编译验证
- 文件职责更清晰，后续维护成本更低
- 更适合作为持续演进版本

### 4.2 查看旧结构

当你需要回看未拆分组织方式时，使用：

- `archive-monolithic`

适用场景：

- 想快速定位拆分前的原始代码集中位置
- 需要人工对照“拆分前后”的结构和调用链
- 想核对某段逻辑是否在拆分时发生位置迁移

### 4.3 做精确回退或对比

当你需要一个稳定锚点时，使用：

- `monolithic-baseline`

适用场景：

- 临时 checkout 到未拆分基线
- 做 `git diff` / `git range-diff` / GitHub compare
- 作为文档、评审或回归说明里的固定引用点

## 5. 常用 Git 操作

### 5.1 切到拆分完成版

```bash
git switch main
```

### 5.2 切到未拆分归档版

```bash
git switch archive-monolithic
```

### 5.3 基于标签查看未拆分基线

```bash
git switch --detach monolithic-baseline
```

### 5.4 对比两套版本

```bash
git diff monolithic-baseline..main -- app/src/main/java/com/poxiao/app/ui
```

### 5.5 查看某个文件在未拆分基线中的内容

```bash
git show monolithic-baseline:app/src/main/java/com/poxiao/app/ui/PoxiaoApp.kt
```

## 6. 后续维护建议

1. `main` 只承载“拆分完成版”的后续开发，不再回退成单文件主结构。
2. `archive-monolithic` 作为历史归档分支，原则上不继续堆叠功能提交。
3. `monolithic-baseline` 保持为固定锚点，不重打、不漂移。
4. 以后如果再有“大规模重构完成且验证通过”的节点，建议继续采用：
   - 主线分支承载当前稳定版
   - 归档分支保留上一代结构
   - 标签固定关键回退点

## 10. 管理规范建议

1. **语义化版本 (SemVer)**：
   - `X` (Major): 架构级变更（如从单体转向多智能体）。
   - `Y` (Minor): 功能级迭代（如本轮的联动增强）。
   - `Z` (Patch): 缺陷修复与小细节优化。
2. **标签规范**：每次 Minor 以上版本发布前，必须在 `main` 打上 `vX.Y.Z-description` 格式的标签。
3. **同步更新**：版本号更新需同步修改 `app/build.gradle.kts` 中的 `versionName` 和 `versionCode`。
