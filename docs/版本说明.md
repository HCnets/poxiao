# 版本说明

## 1. 文档目的

这份文档用于说明当前仓库中“未拆分基线版”和“拆分完成版”的保留方式、适用场景与建议工作流，方便后续开发、回退、对比和交接。

## 2. 当前版本布局

- 主维护线：`main`
- 归档基线分支：`archive-monolithic`
- 未拆分基线标签：`monolithic-baseline`

## 3. 两套版本定义

### 3.1 拆分完成版 (v1.3.3 Unit-Aware Computation)

- 分支：`main`
- 角色：
  - 作为后续开发和维护的主线
  - **单位感知计算 (Unit-Aware)**：支持物理单位的混合运算（如 `5km + 300m`），自动处理量纲转换与推导（如 `m * m = m^2`）。
  - **实时函数图像预览 (Sparkline)**：在显示屏实时渲染函数图像，增强直观感知。
  - **触觉光标刻度感 (Haptic Ticks)**：增强光标移动时的物理反馈感。
  - **自然数学渲染 (WYSIWYG)**：支持嵌套分数、根式、乘方的真实数学排版渲染。

当前特点：

- `PoxiaoApp.kt` 已收缩为入口壳层
- `PoxiaoAppScaffold.kt` 已收缩为宿主壳层
- `HomeScreen.kt` 已收缩为状态装配 + 页面壳层
- 共享模型、动作包、builder、component、helper 的边界已显著清晰化
- 用户已确认 Android Studio 中“编译成功，功能正常”

### 3.2 稳定基线版 (v1.3.2 Intelligence Awareness)

- 分支：`archive-monolithic`
- 角色：
  - 引入实时函数图像预览与触觉光标增强的第一个稳定版本。

### 3.3 历史归档版 (v1.3.1 Natural Math Engine)

- 分支：`archive-monolithic`
- 标签：`monolithic-baseline`
- 基线提交：`7a77a44`
- 提交说明：`revert: restore monolithic poxiao app baseline`
- 角色：
  - 作为拆分前的归档对照版本
  - 用于结构回看、差异比对和应急回退
  - 不建议作为后续日常开发主线

当前特点：

- 结构更接近拆分前的单文件 / 大文件组织方式
- 适合做“前后结构差异”对照
- 适合排查“某逻辑原来在哪个大文件里”

## 4. 建议工作流

### 4.1 日常开发

默认使用：

- `main`

原因：

- 当前这条线已经经过本轮真实编译验证
- 文件职责更清晰，后续维护成本更低
- 更适合作为持续演进版本

### 4.2 查看旧结构

当你需要回看未拆分组织方式时，使用：

- `archive-monolithic`

适用场景：

- 想快速定位拆分前的原始代码集中位置
- 需要人工对照“拆分前后”的结构和调用链
- 想核对某段逻辑是否在拆分时发生位置迁移

### 4.3 做精确回退或对比

当你需要一个稳定锚点时，使用：

- `monolithic-baseline`

适用场景：

- 临时 checkout 到未拆分基线
- 做 `git diff` / `git range-diff` / GitHub compare
- 作为文档、评审或回归说明里的固定引用点

## 5. 常用 Git 操作

### 5.1 切到拆分完成版

```bash
git switch main
```

### 5.2 切到未拆分归档版

```bash
git switch archive-monolithic
```

### 5.3 基于标签查看未拆分基线

```bash
git switch --detach monolithic-baseline
```

### 5.4 对比两套版本

```bash
git diff monolithic-baseline..main -- app/src/main/java/com/poxiao/app/ui
```

### 5.5 查看某个文件在未拆分基线中的内容

```bash
git show monolithic-baseline:app/src/main/java/com/poxiao/app/ui/PoxiaoApp.kt
```

## 6. 后续维护建议

1. `main` 只承载“拆分完成版”的后续开发，不再回退成单文件主结构。
2. `archive-monolithic` 作为历史归档分支，原则上不继续堆叠功能提交。
3. `monolithic-baseline` 保持为固定锚点，不重打、不漂移。
4. 以后如果再有“大规模重构完成且验证通过”的节点，建议继续采用：
   - 主线分支承载当前稳定版
   - 归档分支保留上一代结构
   - 标签固定关键回退点

## 9. 产品线版本矩阵 (Edition Matrix)

针对不同用户群体，项目划分为三条产品线：

| 产品线 | 代号 | 核心定位 | 关键能力 |
| :--- | :--- | :--- | :--- |
| **哈深专版** | `hitsz` | 参赛主版本 | 教务 + 校园服务 + 地图 + AI + 全域联动 |
| **通用教务版** | `academic` | 通用适配版 | 通用教务接口 + AI + 复习/待办联动 |
| **轻量工具版** | `lite` | 纯效率版 | Todo + 番茄钟 + 笔记 + 复习 (无教务依赖) |

### 9.1 Lite 版功能裁剪说明 (Lite Edition Restrictions)

为了保持 Lite 版本的轻量化和隐私安全，系统在运行时会对以下能力进行严格隔离：

1. **入口隐藏**：
   - 底部导航栏与侧边栏自动移除“课表”分区。
   - 更多页面隐藏“教务账号”绑定入口和“校园服务”入口。
2. **UI 降级**：
   - 若用户通过深层链接或其他方式进入“课表”页面，将显示“当前版本未启用教务功能”的优雅占位。
   - 首页“快速检索”自动移除成绩和课程搜索，仅保留待办、笔记和楼栋。
3. **逻辑隔离**：
   - 首页模块（Dashboard）自动过滤“课表”相关卡片，仅保留专注、学习指标（基于待办）和智能助手。
   - 导出中心动态移除“课表摘要”和“成绩趋势”导出项。

### 9.2 发版策略

1. **版本号共用**：所有产品线共享同一套 `SemVer` 主版本号（如 `1.1.0`）。
2. **后缀区分**：通过 Gradle 后缀区分构建包（如 `1.1.0-hitsz`, `1.1.0-lite`）。
3. **能力裁剪**：通过 `EditionCapabilities` 在运行时动态控制 UI 入口和功能逻辑。

## 10. v1.2.0 详细更新日志 (MAS & Liquid Glass 2.0)

### 10.1 多智能体系统 (MAS) 落地
- **专家并行调度**：引入 `AgentOrchestrator`，基于 Kotlin 协程 `async/awaitAll` 实现学务、效率、复习等六大本地专家的并发执行，大幅降低首字响应延迟。
- **Reflection 协议**：引入 DeepSeek 作为反思专家，当本地规则匹配置信度不足时，触发二次润色与逻辑兜底。
- **真实链路打通**：`AcademicExpert` 脱离 Mock 数据，直接挂载至 `AcademicRepository` 获取真实的 HITSZ 课表与教务状态。

### 10.2 Liquid Glass 2.0 视觉重构
- **陀螺仪视差动效 (Parallax)**：重写 `LiquidGlassSurface` 的位移逻辑，加入低通滤波 (Low-Pass Filter) 与基准面校准，实现丝滑、防抖的重力反馈。
- **高阶预设重命名**：将风格预设升级为 `星辉雾晶 (Aura)`、`冰川透镜 (Glacier)`、`流光霓虹 (Prism)`。
- **无级自定义调节**：在偏好设置中暴露 UI 滑动条，支持用户独立调节全局雾度 (Blur)、发光强度 (Glow) 与背景透明度 (Alpha)。
- **可视度增强**：引入动态雾度补偿与内阴影 (Inner Shadow) 算法，确保在任何复杂背景下文字依然清晰可读。

## 12. 管理规范建议

1. **语义化版本 (SemVer)**：
   - `X` (Major): 架构级变更（如从单体转向多智能体）。
   - `Y` (Minor): 功能级迭代（如本轮的联动增强）。
   - `Z` (Patch): 缺陷修复与小细节优化。
2. **标签规范**：每次 Minor 以上版本发布前，必须在 `main` 打上 `vX.Y.Z-description` 格式的标签。
3. **同步更新**：版本号更新需同步修改 `app/build.gradle.kts` 中的 `versionName` 和 `versionCode`。
